# 工程原则

## 目标
建立一套面向当前项目的长期工程思想框架，让 Architect 与 Operator 在设计、实现与审核过程中共享清晰的一致性判断标准，确保职责、边界与扩展策略保持可检测、可追踪、可维护。

## 适用范围
适用于当前项目仓库中所有影响架构、域逻辑、共享模块与基础设施的决定，对任何涉及模块边界、状态控制、依赖注入、输入校验或系统一致性的改动都应以此为首要参考。

## 核心原则
- 小而单一职责：每个函数或模块都应专注于一个明确目标，函数体自然控制在 20-30 行左右，必要时通过拆分或辅助手段维持可读性。
- 组合优于继承：通过接口、策略、适配器或装饰器扩展行为，避免继承链超过两层，尽可能不在老函数里堆砌新的 `if/else`。
- 状态最小化：核心域逻辑保持纯或近纯，所有副作用（缓存、网络、持久化、日志）集中在系统边缘，由边界函数明确触发与管理。
- 显式边界：公共 API、DTO、服务契约与状态转换必须明确类型与接口，所有外部输入（HTTP、CLI、队列、文件）必须在入口处校验。
- 模块隔离：模块之间通过小而稳定的接口或 mapper 通信，避免直接依赖他模块的内部结构，运输模型与领域模型保留各自职责。
- 增量扩展优先：在已有稳定行为上通过新增模块、策略、适配器或映射表进行扩展，不随意重写或膨胀旧逻辑。
- 可验证一致性：每次架构决策都参考本规范、自检清单与 `.trellis/specs/` 目录中的其他长期规则，确保长期可维护性。

## 设计决策偏好
- 在引入新能力时优先新增独立模块、策略实现、适配器或配置驱动的映射表，而非直接膨胀老函数的控制流。
- 将纯域计算与 IO/框架胶水分离，先构建可单元测试的核心逻辑层，再通过边界适配器负责副作用与外部协调。
- 所有依赖通过参数、构造器、工厂等显式注入，杜绝隐式全局或环境魔法式访问。
- 强制使用小而稳定的接口、DTO 或 mapper 作为模块契约，避免一个模块直接“伸手”访问另一个模块的内部结构。
- 明确记录输入、输出与失败路径，确保外部调用方在边界层校验后再进入内层逻辑，避免重复校验。
- 决策应与本目录中的模块边界、代码质量和工作流规范保持一致，避免出现彼此冲突的局部约定。

## 自检问题
- 本次改动是否已经先考虑通过新增模块/策略/适配器/映射表实现，而不是直接膨胀旧逻辑？
- 模块边界、依赖关系、输入与输出契约是否被清晰识别并记录？
- 核心逻辑是否可脱离 IO 单独测试，副作用是否集中在边界适配器？
- 每个函数或模块是否仍只承担一个清晰职责，函数体是否在 20-30 行的合理范围内？
- 是否优先使用组合、接口、策略或适配器，而不是继承链或 `if/else` 堆砌？
- 是否已避免隐藏状态与跨模块隐式耦合，只通过显式数据流传递状态？
- 外部输入（HTTP、CLI、消息、定时任务、持久化等）是否在边界处完成校验？
- 模块之间是否通过稳定接口/DTO/mapper 交互，而非直接依赖彼此内部结构？
- 修改是否保持了原有稳定行为，而是在其基础上通过增量方式引入新能力？
- 以上原则是否与 `03-code-quality-rules.md` 中的自检清单一致？
