# 代码质量规则

## 目标
建立每日编码、修改与提交流程的长期质量基线，使每次提交都保持清晰、可验证、可维护的标准，从而让后续的审查与演进可以在可预测的基础上进行。

## 适用范围
适用于当前项目仓库中所有将被运行、审查或自动化检查的源代码、脚本与 Trellis 相关文档，贯穿编码→修改→提交/文档发布的全过程。

## 日常编码规则
- 保持每个函数或模块只有一个清晰职责，函数体自然控制在 20-30 行，必要时通过合理拆分进一步降低复杂度。
- 把域逻辑与 IO/框架胶水分层组织，核心计算保持可单元测试，所有副作用集中在边界适配器。
- 优先通过组合、接口、策略、适配器或装饰器来构建行为，而不是依赖继承树或在已有函数里叠加分支。
- 避免继承链超过两层，除非框架明确强制，并优先通过策略/策略组合来复用行为。
- 最小化隐藏状态，把缓存、连接、事务等保持在边缘层，并通过显式数据流或 DTO 传递必须的状态。
- 所有公共 API、服务入口、持久化读写、消息/定时任务消费等外部输入必须在边界处完成校验。
- 依赖必须通过参数、构造器或工厂显式注入，避免隐式全局变量或环境魔法式访问。
- 模块之间通过小而稳定的接口、DTO 或 mapper 交互，保持运输模型与领域模型的边界清晰。
- 重要行为、决策点与边界契约应在文档或注释中记录，便于 Operator 审查与后续维护。

## 修改策略
- 优先以新增模块/策略/适配器/映射表来扩展行为，避免直接重写已有稳定逻辑。
- 当需要引入多种变异行为时，优先使用配置、映射表或策略模式，避免在单个函数中堆砌多个 `if/else`。
- 大函数应拆分成小、可测试的单元，每个函数只处理一类变换，保持修改可回退。
- 核心逻辑尽量保持纯或近纯，只有边界适配器负责副作用，再通过显式接口触发。
- 任何关键决策与边界契约都应同步到 Trellis 规范或相关文档中，确保长期一致性。

## 文档与提交要求
- 影响架构、模块边界或本规范原则的改动都必须同步更新 `.trellis/specs/` 中的相关条目。
- 提交说明应描述已完成的自检项（例如原则、边界、组合策略）以及运行过的验证（单元/集成/静态分析等）。
- 在相关文档或注释中记录输入校验、依赖注入、状态控制等关键决策，便于后续 Operator 审核。
- 提交语义清晰，说明变更涉及的模块、策略或适配器，并指向更新后的 Trellis 规范。

## 自检问题
- 编码前
  - 本次改动是否优先通过新增模块/策略/适配器/映射表实现，而不是直接扩张旧逻辑？
  - 是否已经明确本次改动的模块边界、依赖关系与输入/输出契约？
  - 是否能将核心逻辑与 IO/框架胶水分开实现？
- 编码中
  - 每个函数是否保持清晰职责，函数体是否控制在 20-30 行？
  - 是否优先使用组合、接口、策略、适配器，且避免继承链超过两层？
  - 核心逻辑是否保持纯或近纯？隐藏状态与跨模块耦合是否被显式化？
- 边界与契约
  - 公共 API、共享类型与核心状态变换是否明确类型与行为契约？
  - 外部输入（HTTP、CLI、消息、定时任务、持久化）是否在边界处完成校验？
  - 依赖是否通过参数/构造器/工厂显式注入？是否避免隐式全局变量与环境魔法？
- 模块协作
  - 模块之间是否通过小而稳定的接口、DTO 或 mapper 通信，而不依赖彼此内部结构？
  - 业务规则是否与控制器、路由、ORM 客户端、框架配置等胶水代码分离？
- 扩展与修改
  - 是否优先以增量方式扩展稳定逻辑，而不是重写？
  - 是否避免在旧函数里继续堆大型 `if/else` 分支，而是引入新的策略/模块？
  - 本次改动是否仍选择“最简单且安全”的方案，而非为抽象而抽象？
- 完成前
  - 核心函数是否仍然容易理解？
  - 核心逻辑是否可脱离 IO 单独测试？
  - 新增的依赖、边界契约、输入校验是否显式记录？
  - 这次改动是否降低了未来扩展与 AI 继续编辑的难度？
