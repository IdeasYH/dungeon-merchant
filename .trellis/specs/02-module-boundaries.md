# 模块边界

## 1. 目标
- 提炼跨模块、跨团队长期有效的边界认知，便于 Architect 与 Operator 在架构、实现、文档编制等节点保持一致。
- 明确依赖、输入、文档和自检的最低约束，防止模块间耦合、状态泄露和职责模糊的长期风险。

## 2. 适用范围
- 本规范覆盖所有业务模块、基础设施适配层、文档输出面向和跨层接口契约，适用于平台级长期演化。
- 不仅指导代码实现，也约束 Trellis 规范、文档产出与协作沟通中对边界、责任的描述。

## 3. 边界原则
- **单一职责**：每个模块/函数保持 1 个清晰目的，实在需要多步流程时拆成多个具名阶段，避免 20-30 行以上的“混合职责”函数。
- **组合优于继承**：拓展行为时优先引入策略、适配器或 wire-up 映射表，避免继承链超过 2 层，以免引入隐式状态或硬编码决策。
- **状态显式、纯净优先**：核心域逻辑尽量使用纯/近纯函数，副作用（网络、持久层、IO）仅在边缘处理，保持断言容易、测试方便。
- **增量扩展**：遇到稳定逻辑先扩展，新模块/策略/配置优先于重写老逻辑，减少一次性大调整带来的边界不确定性。
- **边界清晰**：每个模块公开的接口、共享类型和状态变换都必须有显式契约，避免对其他模块内部结构的直接依赖。

## 4. 依赖与协作规则
- 模块之间通过小而稳定的接口、DTO 或映射器通信，跨层信息不应泄露实现细节，避免“穿透访问”。
- 依赖注入、构造器参数或显式工厂必须传递模块需要的上下文；禁止依赖全局单例、环境魔法或隐式共享状态。
- 共享责任（例如鉴权、缓存、日志）应封装为独立策略/适配器模块，由调用方组合使用，保持模块互不干扰。
- 任何新的依赖关系需验证是否改变边界契约，如果是，应先在文档中说明更新的接口/版本信息，再落地代码。

## 5. 输入与契约要求
- 所有跨模块输入（HTTP、CLI、消息、持久化读写等）必须在边界处校验，防止“脏”数据漫延到内部逻辑。
- 公共 API 应使用显式类型定义/DTO，确保调用方与提供方对字段、语义、错误边界有共同理解。
- 输入验证、转换、状态变化的结果应明确返回值/异常契约，避免不同实现产生不一致的成功/失败语义。
- 特殊边界（例如配置、特性开关、第三方 client）应提供文档说明，并明确表示依赖的版本、兼容策略与回退路径。

## 6. 文档边界
- 长期规则、模块边界、职责分工以 `.trellis/specs/` 为主目录；临时任务/功能设计另存 `docs/superpowers/specs/` 和 `.trellis/tasks/`，避免长期规范被动态更新覆盖。
- 每个长期规则文档应明确适用面向，避免多个文档表达相同边界而产生冲突；若有变更，优先在本目录统一口径。
- 文档产出也应遵循“边界清晰”原则：规范类文档聚焦职责、架构、契约；实现类文档聚焦特定行为和上下文，使读者在查阅时分清用途。

## 7. 自检问题
- 模块之间是否只通过稳定接口沟通，无漏出内部结构？
- 每个函数/组件是否仍只承担一个显性职责？超过 30 行的逻辑是否拆成了更小单元？
- 依赖是否显式注入，是否避免了全局、隐式状态的泄漏？
- 输入/输出契约是否在边界被校验，公共接口是否提供明确的类型描述？
- 文档是否记录了边界、依赖和版本协议，避免不同模块对同一契约产生不同理解？
