# 前端中文化与认证联调设计

> 日期：2026-04-02  
> 适用仓库：`dungeon-merchant-claude`

## 目标

在不改变接口路径、请求/响应字段、变量名、文件名和路由路径的前提下，完成以下工作：

1. 将当前前端用户可见功能文案调整为中文
2. 保留 `Dungeon Merchant` 等专有名词英文
3. 更新浏览器标题与前端说明文档中的用户可见说明
4. 在开发进度手册中记录本次改动与验证结果
5. 同时完成登录/注册接口的自动化测试与实际联调验证

## 范围

### 包含

- `frontend/src/App.vue` 中的顶栏导航与标题性功能文案
- `frontend/src/views/LoginView.vue` 登录页功能文案
- `frontend/src/views/RegisterView.vue` 注册页功能文案
- `frontend/src/views/HomeView.vue` 主页摘要、状态提示、按钮文案
- `frontend/index.html` 浏览器标题与语言属性
- `frontend/README.md` 前端说明文案
- `docs/superpowers/DEVELOPMENT_STATUS.md` 开发进度记录
- 登录/注册接口的自动化与联调验证

### 不包含

- 接口路径调整
- 请求/响应字段名调整
- 路由路径改名
- 变量名、函数名、文件名改名
- 引入 `vue-i18n` 等国际化框架
- 无关页面重构或视觉改版

## 现状与问题

当前前端基础功能已完成，但主要页面中的用户可见文本仍以英文为主，包括：

- 顶栏导航
- 登录页表单标签、按钮、状态文案
- 注册页表单标签、按钮、状态文案
- 首页摘要卡片与账户信息说明
- 浏览器标题
- 前端 README 初始模板说明

同时，认证接口虽然已有自动化测试，但本轮需求还要求将“自动化测试通过”和“实际接口联调可用”一起作为交付的一部分，并将结果写入开发进度手册。

## 方案选择

本次采用“轻量中文化映射，定点替换文案”的方式。

### 方案说明

- 保持现有组件结构不变
- 直接替换用户可见文案为中文
- 若个别页面存在重复文案，可在文件内提取为局部常量，但不额外引入新的国际化层

### 选择理由

- 改动小，风险低
- 与当前项目规模匹配
- 不会影响接口联调和页面逻辑
- 便于快速完成并验证

## 设计细节

### 1. 前端中文化策略

中文化仅作用于用户可见文案，具体包括：

- 顶栏中的导航按钮和功能按钮
- 页面中的标题、副标题、描述文案
- 表单标签、占位提示、错误提示、加载提示
- 首页账户摘要字段标签

专有名词如 `Dungeon Merchant`、`Guild Ledger`、`Pinia`、`JWT`、接口路径等保持英文，以避免世界观或技术名称被误译。

### 2. 页面处理方式

#### `frontend/src/App.vue`

- 将导航区功能按钮文案改为中文
- 保留品牌主名中的专有名词英文

#### `frontend/src/views/LoginView.vue`

- 保留世界观氛围性文案中的专有名词
- 将登录入口、账号密码、错误提示、提交按钮等功能文案改为中文

#### `frontend/src/views/RegisterView.vue`

- 将注册说明、确认密码、失败提示、提交按钮等改为中文
- 保留世界观品牌名称英文

#### `frontend/src/views/HomeView.vue`

- 将账户信息标签、刷新按钮、退出按钮、状态提示改为中文
- 日期展示由英文区域格式调整为中文区域格式，以保证页面观感一致

#### `frontend/index.html`

- 将 `lang="en"` 改为 `lang="zh-CN"`
- 将页面标题修改为面向中文用户的标题，同时保留品牌名英文

#### `frontend/README.md`

- 将默认 Vite 英文模板说明改为项目当前前端说明的中文版本
- 内容聚焦本项目已有前端功能、启动方式与结构说明

### 4. 认证验证策略

#### 自动化验证

优先复用已有后端集成测试：

- `src/test/java/com/dungeon/merchant/module/auth/controller/AuthControllerIntegrationTest.java`

目标验证：

- 注册接口返回成功
- 登录接口返回成功
- Refresh 接口仍然正常
- 重复注册与错误登录的错误封装保持正常

如前端文案测试需要同步更新，则同时修改：

- `frontend/src/App.spec.js`
- 其他受中文化影响的前端测试文件

#### 实际联调验证

在应用可启动的前提下，通过直接调用接口验证：

- `POST /api/auth/register`
- `POST /api/auth/login`

验证重点：

- 返回 HTTP 状态正常
- 返回体结构与前端现有调用方式匹配
- 注册后的登录流程可连续工作

## 风险与控制

### 风险 1：前端测试文案断言失效

由于现有前端测试包含英文断言，文案改为中文后测试可能失败。

**控制方式：**
- 同步更新断言文本
- 仅更新与中文化直接相关的断言，避免扩大变更面

### 风险 2：将专有名词误翻译

如果品牌名、技术名被误翻，会造成界面风格和文档表达不统一。

**控制方式：**
- 保留 `Dungeon Merchant` 等专有名词英文
- 仅翻译功能性文案

### 风险 3：联调与自动化结果不一致

接口测试通过但运行时环境存在配置差异，可能导致联调失败。

**控制方式：**
- 自动化测试与实际联调都执行
- 在进度手册中明确分别记录结果
